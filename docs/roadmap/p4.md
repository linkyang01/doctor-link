# P4 Roadmap: Automated Diagnosis Pipeline

## Goal

P4 turns Doctor link into an automated project-level diagnosis pipeline.

The pipeline should help a project move from manual evidence packages to repeatable workflows:

1. load a project diagnosis strategy;
2. list and run reproduction steps;
3. run configured test jobs;
4. create before / after diagnostic packages;
5. compare and verify packages;
6. summarize pipeline health;
7. expose project health in the local workbench.

## Delivery Order

P4 should be delivered in small PRs:

- P4.1: strategy loader and validation;
- P4.2: reproduction management;
- P4.3: executable test matrix;
- P4.4: before / after package workflow;
- P4.5: automated comparison and verification;
- P4.6: GitHub Actions and PR integration examples;
- P4.7: project health summary and Web UI panel;
- P4.8: final smoke, docs, audit, and Issue #19 closure.

## Commands Planned

- `doctor-link strategy validate`
- `doctor-link reproduce list`
- `doctor-link reproduce run <id>`
- `doctor-link test run`
- `doctor-link diagnose before`
- `doctor-link diagnose after`
- `doctor-link diagnose compare`
- `doctor-link diagnose verify`

## Expected Outputs

- strategy validation result;
- reproduction evidence;
- executable test result evidence;
- before / after package metadata;
- pipeline summary JSON and Markdown;
- project health JSON and Markdown.

## Boundaries

P4 must remain local-first and repository-file based.

P4 must not:

- publish a release;
- create a GitHub Release;
- publish to PyPI;
- change repository permissions;
- introduce paid cloud services;
- introduce an external account system;
- change Doctor link's core positioning;
- make the local Web UI write back by default.

## Completion Standard

P4 is complete when:

- all P4.1 through P4.8 tasks are implemented or explicitly deferred in the audit;
- CI passes;
- P4 commands are covered by tests or smoke checks;
- README and TODO files are updated;
- `docs/p4-final-audit.md` is added;
- Issue #19 is closed as completed.
