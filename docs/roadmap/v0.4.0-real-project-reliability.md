# v0.4.0 Real-project Reliability

Version 0.4.0 moves automatic solve from isolated single-package demonstrations toward repeatable work on real repositories.

## Delivery slices

### R1: TypeScript and monorepo targeting

- discover npm, Yarn, and pnpm workspace packages;
- require an explicit package when root-level verification is ambiguous;
- execute checks and Codex inside the selected package while keeping branch and cleanliness controls at the workspace Git root;
- reject absolute, missing, and workspace-escaping package paths;
- retain workspace and package identity in every solve receipt.

Acceptance: a broken Node.js package in a multi-package workspace is reproduced and repaired without touching sibling packages, and unsafe package selection is blocked.

### R2: Interrupted-session recovery

- persist `repairing` before invoking the repair provider;
- resume only schema-v2 sessions;
- require fresh explicit approval and the original repair branch;
- revalidate protected acceptance inputs before continuing;
- preserve the original three-round ceiling.

Acceptance: a simulated process interruption leaves a readable receipt and resumes at the next unused round; branch, input, status, and round-limit violations are blocked.

### R3: Real-project benchmark protocol

- define a versioned YAML/JSON scenario manifest;
- isolate receipts by safe scenario ID;
- distinguish actual failed-check reproduction from environment or preflight blocking;
- report expected-status matches, reproduction rate, repair success rate, mean rounds, and duration;
- include the benchmark route in the installed-wheel capability lab.

Acceptance: the benchmark produces machine-readable and Markdown results, returns nonzero for expectation mismatches, and cannot escape its output directory through a scenario ID.

## Release gate

0.4.0 is releasable only after the full test suite, lint, security, package build/install, repository validation, and installed-wheel capability lab pass. At least one disposable TypeScript monorepo repair and one interrupted-session recovery must be retained as validation evidence. PyPI remains a separate publication decision.
