# Doctor link P3-P5 Roadmap

This roadmap expands Doctor link beyond the P2 local diagnostic workbench.

- P3: AI Coding Collaboration Layer
- P4: Automated Diagnosis Pipeline
- P5: Productization and Release Readiness

Each task should be checked off only after the corresponding code, documentation, tests, and CI updates are completed.

---

## P3: AI Coding Collaboration Layer

Goal: make Doctor link a structured bridge between users, diagnostic packages, and AI coding tools.

### P3.1 AI task template system

- [ ] Define AI task template data model.
- [ ] Add templates for bug fix, regression fix, investigation, refactor risk review, and verification-only tasks.
- [ ] Add template fields for evidence IDs, user assertions, reproduction steps, boundaries, forbidden changes, expected outputs, and verification steps.
- [ ] Generate AI task from a selected template.
- [ ] Preserve human-confirmed issue warnings in every template.
- [ ] Add tests for template rendering.
- [ ] Update README.zh-CN.md and README.en.md.

### P3.2 AI tool handoff packages

- [ ] Define AI tool handoff manifest format.
- [ ] Add Codex handoff profile.
- [ ] Add Cursor handoff profile.
- [ ] Add Continue handoff profile.
- [ ] Add Aider handoff profile.
- [ ] Add OpenHands handoff profile.
- [ ] Add generic Markdown/JSON handoff profile.
- [ ] Add `doctor-link handoff <package_dir> --tool <tool>` command.
- [ ] Include ai-task, ai-context, evidence list, user assertions, investigation boundary, and verification checklist.
- [ ] Add tests and CI smoke.

### P3.3 AI repair result intake

- [ ] Define AI repair result schema.
- [ ] Support recording files changed, summary, claimed fix, risks, assumptions, and evidence used.
- [ ] Add `doctor-link ai-result <package_dir>` command.
- [ ] Write `ai-repair-result.json` and `ai-repair-result.md`.
- [ ] Link AI repair results to user assertions and verification steps.
- [ ] Do not accept unsupported AI claims as verified evidence.
- [ ] Add tests and docs.

### P3.4 Multi-turn AI diagnosis context

- [ ] Define diagnosis round data model.
- [ ] Record each AI pass, user correction, evidence addition, and verification attempt.
- [ ] Generate `diagnosis-history.json`.
- [ ] Generate `diagnosis-history.md`.
- [ ] Add command to summarize prior rounds for the next AI task.
- [ ] Preserve unresolved user-confirmed issues across rounds.
- [ ] Add tests and docs.

### P3.5 Human assertion enforcement

- [ ] Add checker that scans AI task and AI repair result for ignored user assertions.
- [ ] Flag AI responses that dismiss human-confirmed problems without evidence.
- [ ] Generate `assertion-compliance-report.md`.
- [ ] Add `doctor-link assertion-check <package_dir>` command.
- [ ] Integrate assertion compliance into verification status.
- [ ] Add tests for unsupported dismissal cases.

### P3.6 Repair risk review

- [ ] Define repair risk checklist.
- [ ] Detect changed files outside investigation boundary when change list is available.
- [ ] Record risk level: low / medium / high / unknown.
- [ ] Generate `repair-risk-review.md`.
- [ ] Add `doctor-link risk-review <package_dir>` command.
- [ ] Add tests and docs.

### P3.7 P3 quality and closure

- [ ] Add P3 end-to-end smoke workflow.
- [ ] Add sample AI handoff package fixture.
- [ ] Add sample AI repair result fixture.
- [ ] Update P2 Web UI to display AI handoff and AI repair result sections.
- [ ] Update TODO.md checkboxes.
- [ ] Close P3 Issue after all P3 tasks are complete.

## P4: Automated Diagnosis Pipeline

Goal: make Doctor link capable of running project-level diagnostic workflows automatically.

### P4.1 Project diagnostic strategy

- [ ] Define `.doctorlink/diagnosis.yml`.
- [ ] Support project type, default commands, evidence rules, verification rules, and excluded paths.
- [ ] Add validation for diagnosis strategy.
- [ ] Add `doctor-link strategy validate` command.
- [ ] Add tests and docs.

### P4.2 Reproduction script management

- [ ] Define reproduction script metadata model.
- [ ] Add `.doctorlink/reproduce.yml`.
- [ ] Support manual, command-based, and test-based reproduction entries.
- [ ] Add `doctor-link reproduce list` command.
- [ ] Add `doctor-link reproduce run <id>` command.
- [ ] Capture reproduction output into diagnostic package evidence.
- [ ] Add tests and docs.

### P4.3 Test matrix automation

- [ ] Extend `.doctorlink/test-matrix.yml` to support executable test jobs.
- [ ] Add test job status model.
- [ ] Add `doctor-link test run` command.
- [ ] Capture test results into `evidence/test-results/`.
- [ ] Link test results to verification checklist.
- [ ] Add tests and docs.

### P4.4 Automated before / after diagnostic packages

- [ ] Define before/after workflow command.
- [ ] Add `doctor-link diagnose before` command.
- [ ] Add `doctor-link diagnose after` command.
- [ ] Ensure package names and metadata are linkable.
- [ ] Automatically preserve before report path for comparison.
- [ ] Add tests and docs.

### P4.5 Automated comparison and verification

- [ ] Add `doctor-link diagnose compare` command.
- [ ] Add `doctor-link diagnose verify` command.
- [ ] Automatically run compare and verify after after-package generation.
- [ ] Generate a pipeline summary report.
- [ ] Never mark success if verification evidence is missing.
- [ ] Add tests and docs.

### P4.6 GitHub Actions and PR integration

- [ ] Add reusable GitHub Actions example workflow.
- [ ] Add PR diagnostic check mode.
- [ ] Upload diagnostic package artifacts.
- [ ] Add optional comment summary for PRs.
- [ ] Add failure mode that points to diagnostic package artifacts.
- [ ] Add docs and CI validation for example workflow syntax.

### P4.7 Project health dashboard foundations

- [ ] Define project health summary model.
- [ ] Track package count, unresolved assertions, failed verifications, and recent regressions.
- [ ] Generate `project-health.json`.
- [ ] Generate `project-health.md`.
- [ ] Add P2 Web UI panel for project health.
- [ ] Add tests and docs.

### P4.8 P4 quality and closure

- [ ] Add P4 end-to-end fixture project.
- [ ] Add pipeline smoke tests.
- [ ] Add broken reproduction / failed test / missing evidence scenarios.
- [ ] Update README.zh-CN.md and README.en.md.
- [ ] Update TODO.md checkboxes.
- [ ] Close P4 Issue after all P4 tasks are complete.
