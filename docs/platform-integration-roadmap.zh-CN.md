# 平台集成路线图

本路线图是 GitHub/AI 工具直连、团队协作和云端治理的候选产品设计，不代表已经承诺的版本或日期。当前权威状态以 `docs/roadmap.md` 和 `docs/project-status.md` 为准。它们独立于当前本地版本推进，不把现有诊断包变成必须联网或必须登录才能使用的格式。

## 总体原则

1. 永久保留无需账号、无需遥测、无需云服务的本地运行模式。
2. 所有外部写操作必须具备明确对象、操作者身份、授权范围和审计记录。
3. 诊断包保持工具中立，不绑定单一 AI 或代码托管平台。
4. 任何证据离开本地前，必须通过隐私导出门禁和完整性检查。
5. 平台差异通过 Adapter/Plugin 承载，核心诊断模型不写入供应商专有逻辑。

## 阶段划分

| 阶段 | 交付结果 | 前置依赖 | 发布门槛 |
| --- | --- | --- | --- |
| P0 集成基础 | 统一连接协议、授权、审计、幂等与恢复机制 | 当前本地版本 | 未预览、未授权不得外部写入 |
| P1 GitHub/AI 直连 | 导入 Issue、PR、CI 证据并向 AI 工具交付受控上下文 | P0 | 默认只读，写操作逐次批准 |
| P2 团队协作 | 共享诊断队列、角色、评论、指派、审批与活动历史 | P0、P1 身份模型 | 工作空间隔离和权限负向测试通过 |
| P3 云端治理 | 企业身份、密钥、留存、地域、审计、备份与灾备 | P2 | 安全评审、威胁建模、灾备演练通过 |

## P0：集成基础

### 建设内容

- 定义仓库、Issue、PR、CI 运行、AI 会话、审批和外部回执的中立数据对象与 JSON Schema；
- 连接凭据不写入诊断包，个人模式使用系统钥匙串，托管模式使用受管密钥服务；
- 定义 `repository:read`、`ci:read`、`issue:write`、`pull_request:write`、`ai_session:create` 等最小权限；
- 统一预览、确认、幂等键、重试、超时、取消、失败恢复和回滚语义；
- 审计记录包含操作者、供应商、动作、目标、请求摘要、结果和外部回执；
- 隐私门禁或完整性校验失败时，不允许发送证据。

### 验收标准

- 接口 Schema 和兼容性样例全部通过；
- 重复提交不会产生重复评论、Issue、PR 或 AI 会话；
- 凭据过期、限流、断网、取消和供应商局部故障均返回可恢复状态；
- 日志、错误和回执中不存在访问令牌与未脱敏正文。

## P1：GitHub 与 AI 工具直连

### GitHub 连接器

- 读取仓库信息、关联 Issue/PR、Checks、Job 日志和诊断制品；
- 从失败工作流、Issue 或 PR 创建本地诊断任务；
- 评论、上传制品、创建 Issue 或更新 PR 前必须展示写入预览；
- 只上传通过隐私门禁且未被排除的证据；
- 保存运行、任务、评论、PR、Issue 和制品标识作为外部回执。

### AI 连接器

- 继续以现有 handoff 包作为工具中立的事实来源；
- Codex/OpenAI 及其他工具通过 Adapter 声明上下文格式、大小限制和结果 Schema；
- 用户审查文件范围、禁止边界和任务要求后，才允许创建 AI 修复会话；
- 将 AI 声称的修复、改动文件、验证步骤和回执写回诊断包；
- AI 返回成功不能直接把问题标记为已验证，仍必须通过 Doctor link 本地断言和证据检查。

### 验收标准

- 只读预览无需任何写权限即可运行；
- 每一次外部写入都有预览、明确批准、本地不可变审计和供应商回执；
- 上传中断后可以使用同一幂等键安全重试；
- 缺少关联测试或人工断言未解决时，AI 结果保持阻断。

## P2：团队协作

### 建设内容

- 建立 `owner`、`maintainer`、`reviewer`、`viewer` 四级基础角色；
- 提供共享诊断任务队列、负责人、优先级、状态、标签、评论和提醒；
- 敏感导出、AI handoff、外部评论和修复关闭支持审批策略；
- 并发编辑采用版本检查和显式冲突，不使用静默的最后写入覆盖；
- 建立不可变活动流和可导出的审计历史；
- 优先交付可自托管版本，并保留离线诊断包导入导出。

### 验收标准

- 角色权限矩阵具备完整的允许和拒绝测试；
- 通过猜测任务、制品或工作空间标识无法越权读取；
- 并发评论、指派和审批不会丢失任何授权操作；
- 用户撤销或会话过期后立即失去权限；
- 客户可以完整导出诊断任务与审计记录，避免数据锁定。

## P3：云端治理

### 建设内容

- 租户隔离、OIDC/SAML 单点登录、SCIM 生命周期、MFA 策略和服务账号；
- RBAC 以及按项目、数据级别和环境扩展的属性策略；
- KMS/HSM 信封加密、密钥轮换、诊断包签名和验签；
- 可配置留存、法律保留、删除凭证、数据地域和客户自管存储；
- 集中下发隐私、导出、Adapter 和 AI 供应商策略；
- 安全事件导出、管理员审计、备份恢复、跨地域切换和灾备目标。

### 验收标准

- 独立威胁建模和渗透测试关闭全部严重及高风险问题；
- 应用、存储、搜索、缓存、日志和后台任务均通过租户隔离测试；
- 完成密钥轮换与吊销演练，历史合法诊断包仍可验证；
- 留存和删除操作产生可核验回执；
- 备份恢复与跨地域恢复达到声明的 RPO/RTO；
- 具备上线、监控、事件响应、状态通知和回滚运行手册。

## 建议交付节奏

1. 集成基础：协议、授权、审计、幂等与只读预览。
2. 受控写入：经逐次批准的 GitHub/AI 写入、回执和恢复。
3. 团队协作 Beta：优先提供可自托管形态。
4. 云端治理：通过安全、隔离、恢复和运营验收后再确定正式版本。

具体语义版本只能在该阶段获得明确授权、任务清单和验收门槛后确定，不能复用已经发布的 v0.2.0-v0.5.1 版本号。

## 当前版本明确不做

- 不要求云账号；
- 不自动推送代码、合并 PR、评论 Issue 或创建 AI 会话；
- 不在后台上传数据或采集遥测；
- 在 P3 验收前不宣称已具备真实企业 RBAC、受管签名或合规认证。
