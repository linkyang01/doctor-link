## Purpose

P5.10 exists to validate the current Doctor link repository from a clean local development environment before any decision to authorize P6.

This phase does not add product functionality. It records the exact validation commands, expected artifacts, failure handling path, and P6 authorization gate.

## Local validation command sequence

Run from the repository root:

```bash
python3 -m venv .venv
source .venv/bin/activate

python -m pip install --upgrade pip
python -m pip install -e .
python -m pip install pytest pytest-cov build ruff

ruff check doctor_link tests scripts
pytest -q --cov=doctor_link --cov-report=term-missing --cov-report=xml
python -m build

bash scripts/validate_doctor_link.sh
bash scripts/e2e_validate.sh "$(pwd)"
```

## Failure triage workflow

If any command fails:

1. Stop the validation sequence.
2. Record the failed command and full error output.
3. Determine whether the failure is environment-only or repository-related.
4. If environment-only, document the local requirement and rerun from a clean shell.
5. If repository-related, open a targeted fix issue or PR.
6. Do not authorize P6 until validation passes.

## P6 authorization gate

P6 may be considered only after this document records a passing local validation run.

If P6 is authorized, the first implementation area must be P6.1 Diagnostic Package Schema v1. P6 must not start with Web platform, cloud service, external account system, paid service integration, marketplace, or ecosystem portal work.
