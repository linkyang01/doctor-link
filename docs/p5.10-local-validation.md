Phase: P5.10 optional local validation record

## Purpose

This document records how to validate Doctor link from a clean local or target delivery environment.

The repository has already completed P0-P5.10 mainline capability work, GitHub Actions cloud validation, cloud self-validation, P6.1-P6.11 planning / schema / specification closure, and P7.0-P7.10 local runtime implementation and final validation closure.

This document does not add product functionality. It records the exact validation commands, expected artifacts, failure handling path, and local run record format.

## Local validation command sequence

Run from the repository root:

```bash
python3 -m venv .venv
source .venv/bin/activate

python -m pip install --upgrade pip
python -m pip install -e .
python -m pip install pytest pytest-cov build ruff

ruff check doctor_link tests scripts
pytest -q --cov=doctor_link --cov-report=term-missing --cov-report=xml
python -m build

bash scripts/validate_doctor_link.sh
bash scripts/e2e_validate.sh "$(pwd)"
bash scripts/p7_runtime_validate.sh "$(pwd)"
```

## Optional strict integrity validation

After generating a local integrity manifest, the target environment may also run strict untracked-file detection:

```bash
doctor-link integrity manifest . --out DoctorReports/integrity-manifest.json
doctor-link integrity verify . DoctorReports/integrity-manifest.json --strict
```

Strict mode blocks files present under the validation root but missing from the manifest.

## Failure triage workflow

If any command fails:

1. Stop the validation sequence.
2. Record the failed command and full error output.
3. Determine whether the failure is environment-only or repository-related.
4. If environment-only, document the local requirement and rerun from a clean shell.
5. If repository-related, open a targeted fix issue or PR.
6. Do not publish, tag, release, or distribute the project until the target-environment validation issue is understood.
