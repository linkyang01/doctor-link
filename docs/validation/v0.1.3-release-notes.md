# Doctor link v0.1.3

Release date: 2026-07-13

Doctor link `v0.1.3` closes reliability gaps found through complex simulation, makes diagnostic-package export privacy-safe and recoverable, and proves every public CLI route against a cleanly installed wheel.

## Highlights

- All 62 public CLI routes are exercised through 70 real command invocations and eight linked complex scenarios.
- Failed reproduction, required tests, incomplete verification, and blocked diagnosis now return trustworthy non-zero statuses while retaining structured JSON.
- Concurrent package writes use cross-process locks and atomic replacement so evidence records are not silently lost.
- Before/after diagnosis preserves human assertions and requires linked passing evidence before candidate verification.
- `doctor-package` applies the privacy export gate by default and records any explicit reviewed override.
- Export manifests and package README files contain portable relative paths rather than local absolute paths.
- Zip export uses capacity preflight and atomic replacement, preserving the previous complete archive after interruption or insufficient disk space.
- Legacy export-shaped `manifest.json` files migrate safely through `schema migrate`, with backups and formal-manifest protection.
- English and Chinese roadmaps define later GitHub/AI, team collaboration, and cloud-governance stages without changing the local-first boundary.

## Safer diagnosis automation

Commands that produce an unsuccessful machine-readable result also return a non-zero shell status:

```bash
doctor-link reproduce run <id> . --package-dir <package_dir> --json
doctor-link test run . --package-dir <package_dir> --json
doctor-link verify <package_dir> --write-back --json
doctor-link diagnose verify <after_package> --json
```

Optional test jobs remain informational when they fail, but required jobs block the command. Automated evidence is written back with stable rerun identifiers, assertion links, timeline records, and transaction-safe updates.

## Privacy-safe package export

```bash
doctor-link doctor-package <package_dir> --out package.zip --json
```

The command scans only files selected for export. Sensitive findings block the archive by default. Files excluded through attachment, log, screenshot, or size filters are neither archived nor treated as export blockers.

An authorized reviewer can explicitly accept remaining findings:

```bash
doctor-link doctor-package <package_dir> --out reviewed.zip \
  --allow-unsafe-export --json
```

The override and finding count are recorded in `package-export-manifest.json`. The archived manifest and package README do not expose local source or destination paths.

## Legacy export migration

```bash
doctor-link schema migrate <package_dir> --json
```

The command recognizes legacy export-shaped `manifest.json`, creates `package-export-manifest.json`, and preserves the original manifest and README as local backups. It refuses formal package manifests instead of guessing. Migration backups are excluded from subsequent archives.

## Packaging and installation

Download the wheel or source archive from this GitHub Release, then install the wheel:

```bash
python -m pip install doctor_link-0.1.3-py3-none-any.whl
doctor-link --version
doctor-link preflight . --json
```

Install from source:

```bash
git clone https://github.com/linkyang01/doctor-link.git
cd doctor-link
git checkout v0.1.3
python -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
python -m pip install -e .
doctor-link --version
```

Windows PowerShell activation:

```powershell
.\.venv\Scripts\Activate.ps1
```

PyPI publication is not part of this release.

- Supported Python: 3.10, 3.11, and 3.12.
- Validated operating systems: GitHub-hosted Ubuntu, macOS, and Windows.
- Existing local-first diagnostic packages remain supported.
- Legacy export manifests are accepted with a migration warning and can be upgraded automatically.

## Security and privacy

- Diagnostic data stays local unless a user intentionally shares it.
- External export is privacy-gated by default.
- Adapter and Plugin execution remains dry-run by default and requires explicit `--allow-run` approval.
- Unsafe export requires explicit `--allow-unsafe-export` review and leaves an audit field in the archive manifest.
- The release does not add background uploads, telemetry, hosted accounts, or cloud synchronization.

## Known boundaries

This release does not provide:

- a hosted web platform or cloud synchronization;
- external account management or production RBAC;
- automatic GitHub writes or AI session creation;
- production signing-key or KMS infrastructure;
- hosted enterprise archive or diagnostic knowledge base;
- customer-environment or production-acceptance certification.

See `docs/platform-integration-roadmap.md` and `docs/platform-integration-roadmap.zh-CN.md` for the planned separation between local capability, external integrations, team collaboration, and cloud governance.
